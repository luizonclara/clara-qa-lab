Fundamentos de QA

SDLC, STLC, papel do QA, tipos de teste, pirâmide
—
▶

SDLC vs STLC

SDLC = ciclo de vida do software inteiro (requisitos → design → dev → teste → deploy → manutenção). STLC = ciclo de vida só da parte de teste, que vive dentro do SDLC. 🧠 O STLC é um "zoom in" na fase de testes. QA bom atua desde os requisitos (shift-left).

Tipos de teste

NívelO que testaQuem fazVelocidade
UnitárioFunção isoladaDev⚡ rápido
IntegraçãoUnidades juntas / APIsDev/QAmédio
SistemaApp inteiro (visão do usuário)QAlento
AceitaçãoAtende ao negócio?QA/POlento

Também existem: funcional (o que faz) × não-funcional (quão bem: performance, segurança, usabilidade); regressão (garante que mudança nova não quebrou o que já existia); smoke (o build "respira"?) e sanity (checagem rápida e focada após um fix).

Pirâmide de testes

      /\
     /  \     E2E/UI  ← poucos, lentos (Playwright)
    /----\
   /      \   Integração/API  ← médio (Postman)
  /--------\
 /          \ Unitário  ← muitos, rápidos
/____________\

🧠 Anti-padrão = "cone de sorvete" (muito E2E, pouco unitário): lento, instável e caro.

Técnicas de design de teste

partições, valor limite, tabela de decisão, estados
—
▶

Partição de equivalência: divida as entradas em grupos que se comportam igual; teste 1 valor de cada. Valor limite (boundary): bugs moram nas bordas — teste o valor antes, o exato e o depois. Tabela de decisão: para combinações de condições. Transição de estados: para fluxos com estados, testando transições válidas e inválidas.

Exemplo (cupom): "até R$50 sem desconto; de R$50 a R$200 = 10%; acima de R$200 = 15%". Note o problema: R$50 e R$200 estão em duas faixas ao mesmo tempo — ambiguidade clássica de borda. Achar isso é trabalho de QA.

Plano de teste, casos & bug report

os artefatos do dia a dia
—
▶

Plano de teste: o que será (e não será) testado, como, quem, quando, riscos e critérios. Caso de teste: ID · título · pré-condição · passos · resultado esperado · obtido · status. Bug report: título claro, passos pra reproduzir, esperado vs obtido, ambiente, evidência, severidade × prioridade.

🧠 Severidade ≠ Prioridade. Severidade = impacto técnico; Prioridade = urgência de negócio. Ex.: typo no logo = severidade baixa, mas prioridade alta no dia do lançamento.

—
▶

Automação web com Playwright

o carro-chefe do portfólio
—
▶

No dia a dia, o QA automatiza os fluxos críticos (login, compra) que rodam a cada deploy = regressão automatizada; roda em CI; gera relatório e trace.

npm init playwright@latest   # cria o projeto
npx playwright test          # roda
npx playwright codegen URL   # GRAVA suas ações e gera código
npx playwright show-report   # abre o relatório
import { test, expect } from '@playwright/test';
test('login válido', async ({ page }) => {
  await page.goto('https://exemplo.com/login');
  await page.getByLabel('Email').fill('user@teste.com');
  await page.getByLabel('Senha').fill('senha123');
  await page.getByRole('button', { name: 'Entrar' }).click();
  await expect(page.getByText('Bem-vindo')).toBeVisible();
});

Prefira getByRole/getByLabel/getByText (mais estáveis que CSS/XPath). Page Object Model (POM): cada página vira uma classe — organiza testes grandes.

📍 Onde estudar este bloco

  • YOUTUBEStart QA — Playwright + TypeScript (grátis)
  • UDEMYPlaywright + TypeScript — Básico
  • DOCDocumentação oficial · praticar em SauceDemo
🛠️ Pratique junto (opcional): instale o Playwright, use codegen pra gravar um login no SauceDemo, e depois reescreva na mão com getByRole. Esse é o exercício que mais vale no portfólio.

o que te torna avançado(a) e contratável
—
▶

Quando você dominar os blocos, o passo que realmente te contrata é juntar tudo num projeto: testar um e-commerce de demonstração (SauceDemo) com plano de teste + cenários BDD + testes de API (Postman) + testes E2E (Playwright) num repositório público no GitHub. Isso vira seu portfólio.
